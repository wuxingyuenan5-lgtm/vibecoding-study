# 第 1 课教学讲义：用贪吃蛇理解一个前端项目

## 一、本课不是单纯学做游戏

贪吃蛇只是一个规模较小、反馈直接的训练项目。本课真正训练的是：

```text
需求 -> 文件拆分 -> 状态设计 -> 交互实现 -> 运行验证 -> Debug -> Git 提交
```

## 二、三个代码文件的职责

### `index.html`

负责页面结构，例如：

- 游戏标题
- 分数显示
- 游戏区域
- 开始按钮
- 重新开始按钮
- 游戏状态提示

### `style.css`

负责视觉和布局，例如：

- 页面居中
- 游戏区域尺寸
- 按钮样式
- 分数和提示文字
- 蛇、食物、背景的视觉区分

### `script.js`

负责游戏逻辑，例如：

- 保存蛇的位置
- 保存食物位置
- 监听方向键
- 定时移动
- 判断是否吃到食物
- 判断撞墙或撞到自己
- 更新分数
- 开始和重新开始

## 三、五个核心概念

### 1. 状态 State

状态是程序当前保存的数据。本项目至少包括：

- 蛇身体各节的位置
- 当前移动方向
- 食物位置
- 当前分数
- 游戏是否运行
- 游戏是否结束

### 2. 事件 Event

事件是外部操作触发的信号，例如：

- 按下方向键
- 点击开始按钮
- 点击重新开始按钮

### 3. 游戏循环 Game Loop

程序按照固定时间间隔重复执行：

```text
读取方向 -> 计算新位置 -> 判断碰撞 -> 更新状态 -> 重新显示
```

### 4. 渲染 Render

渲染是把内存中的游戏状态显示到页面。每次蛇移动后，都需要根据最新状态重新绘制。

### 5. 碰撞检测 Collision Detection

第一版只判断两类失败：

- 蛇头超出游戏区域
- 蛇头与自身身体重合

## 四、为什么第一版不用框架

本课使用原生 HTML、CSS 和 JavaScript，是为了减少依赖和抽象层，让你直接看到页面结构、样式和逻辑之间的关系。此阶段引入 React、Vue、后端或数据库只会增加无关复杂度。

## 五、与 AI 协作的正确顺序

在让 Codex 写代码之前，先要求它回答：

1. 准备创建或修改哪些文件？
2. 每个文件负责什么？
3. 哪些状态需要保存？
4. 哪些功能属于第一版，哪些明确不做？
5. 最可能破坏什么？
6. 应如何验证？

只有计划合理，才进入编码。

## 六、本课完成标准

完成本课后，你应能解释：

- 为什么项目需要三个文件。
- 蛇的位置为什么属于状态。
- 方向键为什么属于事件。
- 游戏为什么需要循环。
- 验收标准和“看起来能用”有什么区别。
