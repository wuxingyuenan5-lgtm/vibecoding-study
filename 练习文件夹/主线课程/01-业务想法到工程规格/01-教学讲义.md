# 第 1 课教学讲义：从大愿景到可执行小任务

## 一、本课真正要解决的问题

学习者已经有三个真实而且规模很大的方向：

1. 将幕布笔记逐步迁移到 Obsidian，并建设 AI 投研知识库。
2. 继续开发 `wuxingyuenan5-lgtm/news-stugy`，完成信息自动搜集、整理和研究日报。
3. 持续维护 `wuxingyuenan5-lgtm/Platform_Experiment` 基金与交易研究平台。

这些方向都是真实需求，但都不能直接变成一条给 AI 的执行指令。

下面这句话只能作为长期愿景，不能作为工程任务：

```text
帮我把投研全面平台化和 AI 化。
```

它没有说明：

- 当前只处理哪个产品。
- 当前只推进哪个阶段。
- 允许修改哪些文件。
- 输入和输出是什么。
- 明确不做什么。
- 如何验证完成。
- 出错时如何回滚。

因此，本课训练的不是写代码，而是建立正确的任务层级和范围控制能力。

## 二、必须区分的五个层级

### 1. 长期愿景 Vision

长期愿景描述未来希望形成的整体能力。

```text
形成平台化、自动化、AI 驱动的个人投研工作体系。
```

愿景用于决定方向，不直接执行。

### 2. 产品项目 Product

产品项目是承载愿景的独立系统。

```text
A. Obsidian / AI 投研知识库
B. news-stugy 信息自动搜集整理
C. Platform_Experiment 基金与交易研究平台
```

三个项目可以互相连接，但不能在同一任务中混写。

### 3. 阶段目标 Milestone

阶段目标是一个产品在一段时间内要形成的完整能力。

Obsidian / AI 投研知识库可以拆成：

```text
阶段 1：确定 Markdown、YAML、目录和附件规范
阶段 2：完成少量幕布样例迁移
阶段 3：建立检索和索引
阶段 4：接入 AI 问答并提供引用
阶段 5：扩大迁移规模、备份和长期维护
```

`news-stugy` 可以拆成：

```text
阶段 1：稳定 RSS / JSON 输入和日报生成
阶段 2：扩大官方来源并提高去重和事件融合质量
阶段 3：完善调度、日志和失败恢复
阶段 4：接入 LLM Analyzer
阶段 5：建立质量评估和人工复核闭环
```

阶段目标仍然可能包含多个任务，通常不能一次提交完成。

### 4. 课程练习 Exercise

课程练习只服务于一个学习目标。

例如学习 Markdown 和 YAML 时，可以只做：

```text
建立三份模拟投研笔记
→ 比较 YAML 元数据和 Markdown 正文的分工
→ 人工检查结构是否一致
```

这个练习不需要立即迁移真实幕布数据，也不需要接入 Obsidian 插件、数据库或 AI。

### 5. 单次执行任务 Task

单次执行任务是交给 AI 执行端的最小正式工作合同。

```text
只在指定练习目录中新增一个 README 和三份模拟 Markdown 笔记。
每份笔记使用相同 YAML 字段和明确的 Markdown 正文结构。
不读取真实幕布数据，不修改任何真实产品仓库。
```

单次任务必须可执行、可审查、可回滚。

## 三、五个层级之间的关系

```text
长期愿景
  ↓ 选择一个产品
产品项目
  ↓ 确定当前阶段
阶段目标
  ↓ 提取需要学习的能力
课程练习
  ↓ 切成一次可审查改动
执行任务
```

最常见的错误，是从长期愿景直接跳到执行任务。

结果通常是：

- AI 自己补全大量未说明的需求。
- 一次创建过多目录和依赖。
- 修改范围失控。
- 功能很多但无法验收。
- 旧功能被意外破坏。
- 对话和维护成本越来越高。

## 四、能力主线与真实产品的关系

课程不要求每一节课都给真实产品增加功能。

### 能力主线

课程首先训练可迁移能力：

- Git、终端和环境。
- Python、JavaScript 和 Web 基础。
- Markdown、YAML、JSON、SQL 和文件处理。
- API、错误处理和异步任务。
- Debug、测试和 CI。
- 模块、架构、安全和部署。
- LLM、RAG 和 Agent 工作流。

### 产品产出线 A：Obsidian / AI 投研知识库

适合承载：

- 开放格式。
- 元数据规范。
- 幕布迁移。
- 附件管理。
- 搜索与索引。
- AI 检索、问答和引用验证。

### 产品产出线 B：news-stugy

当前已经存在 RSS / JSON 输入、SQLite 存储、规则分析、去重、事件融合、排序和 Markdown 日报等基础。

适合承载：

- Python 和 CLI。
- RSS、JSON 和 HTML 处理。
- SQLite 和数据模型。
- 幂等、去重和事件聚类。
- 测试与 GitHub Actions。
- 调度、日志和失败恢复。
- LLM Analyzer 和质量评估。

已有功能不能因为上课而无意义重写。

### 高级案例库：Platform_Experiment

该项目已经涉及前端、后端、执行运行时、正式账务、风险、对账、测试和部署等复杂模块。

基础阶段主要用于：

- 阅读目录和权威文档。
- 理解模块边界。
- 看懂复杂 PR 和 Diff。
- 学习幂等、审计、Fail-closed、Kill Switch 等工程思想。
- 分析测试和 Release Gate。

默认不直接修改。真实变更必须使用更严格的授权和验证门槛。

## 五、如何判断一项练习是否进入真实产品

使用四个问题判断：

1. 当前能力是否已经讲清？
2. 任务能否独立验收？
3. 失败是否容易回滚？
4. 产品是否真的需要这项改动？

只要有一项回答为“否”，优先在学习仓库中做独立练习。

例如还不理解 SQL、Migration 和事务时，不应该直接修改复杂平台数据库。为了练习某项技术，也不能把不必要的依赖加入真实产品。

## 六、一个合格工程规格的核心部分

### 1. 用户与问题

不合格：

```text
做一个更智能的投研系统。
```

较合格：

```text
同时承担研究和交易的个人用户，需要将分散笔记保存为可迁移文件，以便在 Obsidian 和 AI 工具中继续使用。
```

### 2. 目标

目标描述本轮完成后产生的可观察变化。

```text
完成后，三份模拟笔记使用统一的 YAML 元数据和 Markdown 正文结构保存，可以通过 GitHub Diff 人工核对一致性。
```

### 3. Scope

Scope 是本轮明确包含的工作。

```text
- 一个练习说明 README
- 三份模拟 Markdown 笔记
- 统一的 title、createdAt、tags 元数据
- 每份笔记包含明确的 Markdown 正文
```

### 4. Non-goal

Non-goal 是本轮明确不做的内容。

```text
- 不迁移真实幕布数据
- 不编写迁移工具
- 不安装 Obsidian 插件
- 不接入 LLM
- 不处理附件和目录层级
- 不修改 news-stugy 或 Platform_Experiment
```

Non-goal 不是永远不做，而是当前不做。

### 5. 输入与输出

即使本轮不编写程序，也要明确输入和输出。

```text
输入：三条人工编写的模拟投研内容
输出：三份结构统一、可直接阅读的 Markdown 文件
```

### 6. 边界情况

例如：

- YAML 起止分隔符缺失。
- title 为空。
- createdAt 日期格式不一致。
- tags 被写成普通字符串而不是列表。
- 正文为空。
- 三份文件字段命名不一致。

### 7. 验收标准

验收标准尽量包含：

```text
前提 + 操作 + 预期结果
```

例如：

```text
打开任意一份样例文件时，文件顶部应有完整 YAML Front Matter，分隔线后应出现实际 Markdown 正文。
```

## 七、如何把任务切小

### 规则 1：一次只训练一个主能力

学习 YAML 时，不同时加入数据库、Web 页面、AI 和部署。

### 规则 2：优先切垂直小闭环

```text
准备模拟内容 → 按统一结构保存 → 人工检查结果
```

### 规则 3：一次只承担一种主要风险

不要在一个 PR 中同时修改数据库、交易执行、权限、部署和大型 UI。

### 规则 4：限制文件范围

任务卡必须明确：

- 允许新建什么。
- 允许修改什么。
- 禁止触碰什么。

### 规则 5：先定义验证，再执行

如果无法说明如何验证，说明任务尚未拆清。

## 八、针对三个真实方向的正确拆法

### Obsidian / AI 投研知识库

错误任务：

```text
把幕布全部迁移到 Obsidian，并建立 AI 知识库。
```

合理拆分：

```text
任务 A：设计模拟笔记的 Markdown / YAML 规范
任务 B：建立三份样例并人工检查
任务 C：编写小型字段校验脚本
任务 D：单独处理目录层级
任务 E：单独处理附件相对路径
任务 F：迁移五条脱敏真实样例
任务 G：建立关键词检索
任务 H：设计 RAG 测试集
```

### news-stugy

错误任务：

```text
把所有新闻都自动抓取并用 AI 分析。
```

合理拆分：

```text
任务 A：新增一个官方 RSS 来源并补充解析测试
任务 B：为单个来源失败增加清晰日志
任务 C：为重复 URL 增加幂等测试
任务 D：设计 LLM Analyzer 的结构化输出 Schema
任务 E：使用固定样例评估事实抽取质量
```

### Platform_Experiment

错误任务：

```text
顺便优化基金平台的架构和交易功能。
```

合理学习方式：

```text
任务 A：只读绘制前端、后端和 Runtime 的调用关系
任务 B：选择一个 PR，解释其改动范围和风险
任务 C：阅读一项 Release Gate，说明每个测试防什么错误
任务 D：在独立沙盒复现幂等或对账思想
```

## 九、本轮课堂示范：层级正确，但范围必须对齐

学习者给出的任务地图是：

```text
当前阶段目标：
完成少量幕布笔记的可验证迁移，确认正文、层级、标签和附件能够保留。

本课练习：
使用三份模拟投研笔记，理解 Markdown 正文与 YAML 元数据的分工。

单次执行任务：
在学习仓库的指定目录中新增三份模拟 Markdown 笔记，
每份包含 title、createdAt、tags 三个 YAML 字段；
不编写迁移工具，不修改真实产品仓库。
```

### 判断结果

这三个层级的大小已经基本正确：

- 阶段目标面向未来真实迁移。
- 课程练习只训练 Markdown 与 YAML。
- 单次任务限制为三份模拟文件。

### 需要修正的第一点：阶段目标大于一次任务

“正文、层级、标签、附件”是四个不同验证维度。

当前任务只验证：

```text
YAML 元数据
+ Markdown 正文
```

它不能证明目录层级和附件已经被保留。后两项应分别建立独立任务。

### 需要修正的第二点：YAML 字段不等于完整笔记

只定义 `title`、`createdAt`、`tags`，只能看到元数据，无法真正理解元数据与正文的分工。

每份样例还应至少包含：

```markdown
## 核心观点

模拟正文……

## 证据与风险

模拟正文……
```

### 需要修正的第三点：主动不做也是正确设计

本轮不处理层级和附件，不是遗漏，而是 Scope 控制。

正确顺序可以是：

```text
任务 1：YAML 与 Markdown 正文
任务 2：目录与标题层级
任务 3：附件相对路径
任务 4：少量脱敏真实迁移
```

### 建议采用的单次任务表述

```text
在学习仓库指定练习目录中新增一个 README 和三份模拟投研 Markdown 笔记。

每份笔记：
- YAML Front Matter 包含 title、createdAt、tags；
- Markdown 正文至少包含两个二级标题和对应模拟内容；
- 三份笔记使用不同主题和标签，便于人工比较。

本轮不编写迁移工具，不处理真实幕布数据，不处理附件和目录层级，不接入 Obsidian 插件或 AI，不修改 news-stugy 和 Platform_Experiment。
```

## 十、常见错误

1. 把真实产品当作课程目录。
2. 认为没有产品功能产出就没有学习。
3. 每次都要求 AI 扫描整个大型仓库。
4. 使用“平台化”“AI 化”“提高效率”等无法验收的表述。
5. 一次改动跨越采集、存储、AI、交易和 UI 多个系统层。
6. 把 AI 的完成报告当作验证证据。
7. 用阶段目标描述一次任务，导致任务边界失控。

## 十一、在线审查与本地验证

### 在线可以完成

- 阅读文件结构和文档。
- 检查任务卡和 Scope。
- 检查 Diff 是否超范围。
- 核对三份 Markdown 的字段和正文结构。
- 检查明显的格式错误。

### 通常需要本地完成

- 文件能否被 Obsidian 正确识别。
- YAML 是否被工具正确解析。
- 附件相对路径是否有效。
- 脚本或命令是否真正运行。
- 文件路径、编码和操作系统相关问题。

本轮只建立样例文件时，GitHub 静态审查可以完成主要验收；真正的 Obsidian 兼容性仍应标记为待本地验证。

## 十二、本课术语

| 术语 | 本课含义 |
|---|---|
| Vision | 长期方向，不直接执行 |
| Product | 独立产品或项目 |
| Milestone | 一个阶段形成的完整能力 |
| Exercise | 服务于学习目标的小练习 |
| Task | 一次可执行、可审查、可回滚的改动 |
| Scope | 本次明确包含的工作 |
| Non-goal | 本次明确不做的工作 |
| Acceptance Criteria | 判断完成与否的可执行标准 |
| Vertical Slice | 同时包含最小输入、处理和输出的小闭环 |
| Product Track | 课程中持续产生真实产品增量的路线 |
| Reference Project | 主要用于阅读和审查训练的复杂项目 |
| Front Matter | Markdown 文件顶部由分隔线包围的结构化元数据 |

## 十三、本课结论

学习者的长期产品规划不需要缩小，但每次执行任务必须缩小。

```text
愿景可以很大
阶段必须清楚
课程只练少数能力
任务必须足够小
验收必须有证据
```
