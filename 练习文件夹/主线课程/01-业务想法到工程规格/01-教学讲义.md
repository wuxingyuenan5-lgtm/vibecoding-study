# 第 1 课教学讲义：从业务想法到工程规格

## 一、为什么第一课不是写代码

Vibe Coding 最常见的失败，不是模型不会写代码，而是输入给模型的任务本身不完整：

- 目标不清楚。
- 功能范围不断变化。
- 字段和状态没定义。
- 只描述“长什么样”，没描述“怎么工作”。
- 没有验收标准，最后只能凭感觉判断。

对有明确业务想法、但工程经验不足的人来说，最重要的能力不是背语法，而是把业务语言翻译成工程语言。

```text
模糊想法：做一个可以配合 AI 使用的投研知识库

工程规格：谁使用、解决什么问题、数据如何进入、以什么格式保存、如何检索和修改、附件怎么处理、第一版不做什么、如何验收
```

## 二、业务语言和工程语言的区别

### 业务语言

业务语言强调价值和结果：

- 希望系统自动整理每日信息。
- 希望笔记可以夹杂各种格式的资料。
- 希望摆脱封闭软件，方便迁移到 Obsidian 或其他工具。
- 希望后续建立 AI 知识库并使用 AI 能力。

这些表达是真实需求，但尚不能直接交给 AI 编码。

### 工程语言

工程语言必须继续明确：

- 信息来自哪些渠道。
- “自动整理”包括采集、去重、分类、摘要还是归档。
- 支持哪些附件格式。
- 附件需要上传、预览、全文检索还是只保存引用。
- 笔记使用 Markdown、数据库记录还是其他格式保存。
- 如何导出以及导出后能否继续使用。
- 第一版包含哪些动作。
- 什么条件下算完成。

业务目标回答“为什么做”，工程规格回答“具体做成什么”。

## 三、一个完整规格的七层结构

### 1. 用户与问题

先明确主要用户，不要一开始试图满足所有人。

本课已经确定核心用户：

```text
同时承担研究和交易的个人用户
```

这个角色的特点是：

- 既需要吸收大量外部信息，又需要沉淀自己的判断。
- 研究和交易之间需要形成可回看的闭环。
- 没有专门的信息管理团队，系统必须适合个人维护。
- 更重视数据所有权、迁移能力和 AI 可访问性。

### 2. 目标

目标应描述第一版带来的可观察变化。

结合学习者当前需求，第一版目标可暂时表述为：

```text
让同时承担研究和交易的个人用户，能够以开放、可迁移的格式新增、查看、编辑、检索和导出投研笔记，为后续接入 AI 知识库和自动信息整理打下基础。
```

注意：这里强调“打下基础”，不是第一版就完成自动信息采集和 AI 研究系统。

### 3. 范围与非目标

范围决定本次做什么；非目标防止开发过程中不断加需求。

第一版建议包含：

- 新增笔记。
- 查看笔记列表和详情。
- 编辑已有笔记。
- 关键词搜索。
- 使用开放格式保存或导出。
- 保存附件或外部资料的引用信息。

第一版暂不包含：

- 每日信息自动采集。
- 多渠道爬取或 RSS 聚合。
- AI 自动分类、摘要和问答。
- PDF、图片、音视频的全文解析与 OCR。
- 实时行情。
- 云数据库和多端同步。
- 登录、多人协作和权限管理。
- 完整迁移幕布全部历史数据。

非目标不是永远不做，而是本轮不做。

## 四、需求必须拆成不同系统层

学习者提出的三个问题实际上不属于同一个功能模块，而是三个层次。

### 1. 信息进入层

解决：

```text
每日信息如何自动进入系统
```

可能涉及：

- RSS、邮件、网页、API、聊天记录或人工转发。
- 定时任务。
- 去重。
- 来源记录。
- 自动分类和摘要。

这是后续 API、异步任务和 Agent 工作流课程的内容。

### 2. 知识库存储层

解决：

```text
资料以什么结构保存，如何检索、修改、引用和迁移
```

可能涉及：

- Markdown 文件。
- YAML Front Matter 元数据。
- 普通文件夹。
- 附件相对路径。
- 标签、主题、标的和日期。
- 全文搜索。

这是当前第一版应优先建立的基础。

### 3. AI 应用层

解决：

```text
AI 如何读取、检索、总结、问答和生成研究结论
```

可能涉及：

- 文档切分。
- 向量检索。
- RAG。
- 权限和上下文控制。
- 引用来源。
- 结果验证。

如果存储层封闭、格式混乱，AI 应用层会非常难做。因此当前应先解决数据结构与开放格式问题。

## 五、用户流程

本课案例的最小主流程可以是：

```text
打开知识库
→ 查看笔记列表
→ 点击新增
→ 填写标题、主题、正文、来源和标签
→ 保存
→ 在列表中看到新记录
→ 通过关键词找到记录
→ 打开详情
→ 修改内容
→ 导出或在其他 Markdown 工具中继续使用
```

流程应覆盖正常路径，也要考虑失败和空状态。

## 六、数据字段与数据模型

字段不是越多越专业。第一版只保留能支持核心流程的字段。

建议字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `id` | 唯一标识 | 区分每条记录 |
| `createdAt` | 日期时间 | 记录创建时间 |
| `updatedAt` | 日期时间 | 记录最后修改时间 |
| `title` | 文本 | 笔记标题 |
| `topic` | 文本或枚举 | 标的、行业、宏观主题或策略主题 |
| `content` | Markdown 长文本 | 笔记正文 |
| `source` | 文本或链接 | 原始信息来源 |
| `tags` | 文本数组 | 辅助分类和检索 |
| `attachments` | 引用列表 | 记录附件名称、路径、类型和来源 |

后续可以增加：

- 观点状态。
- 催化剂。
- 风险。
- 关联标的。
- 交易动作。
- 置信度。

但这些是否属于第一版，仍需根据核心使用场景判断。

## 七、开放格式和可迁移性

“可以导出”仍然太模糊，必须明确导出后得到什么。

对于希望兼容 Obsidian 或其他知识库工具的个人用户，一个较稳妥的方向是：

```text
Markdown 正文
+ YAML Front Matter 元数据
+ 普通文件夹结构
+ 附件使用相对路径
```

示例：

```markdown
---
title: 铜价供需研究
topic: 铜
tags:
  - 金属
  - 宏观
createdAt: 2026-07-23
source: https://example.com
---

# 核心观点

正文内容……
```

这样做的价值：

- 人可以直接阅读。
- Git 可以追踪差异。
- Obsidian 等工具可以直接使用。
- AI 工具较容易解析。
- 即使原系统停止维护，数据仍然存在。

但需要注意：

- “兼容 Obsidian”不等于依赖 Obsidian 专有插件。
- 附件路径必须可迁移。
- 文件名需要处理特殊字符和重复名称。
- 元数据字段需要稳定，不能频繁改名。

## 八、混合格式文件的范围

“可以夹杂其他格式文件”至少有四种不同实现：

1. 只保存外部文件链接。
2. 把附件复制到知识库目录，并在笔记中引用。
3. 在页面内预览 PDF、图片、表格等附件。
4. 对附件内容进行全文检索和 AI 解析。

复杂度逐级上升。

第一版建议只做到：

```text
保存附件或资料的名称、类型、来源链接或相对路径，并能从笔记详情中打开。
```

PDF 解析、OCR 和音视频转写应留到后续。

## 九、核心操作的取舍

学习者第一轮选择了：

- 新增笔记。
- 查看列表。
- 查看详情。
- 导出。
- 关键词搜索。

这些方向基本正确，但存在两个需要修正的地方。

### 1. 编辑应进入第一版

研究笔记会持续更新。如果不能编辑，笔记只能一次性写入，无法记录观点修正、补充证据或更新结论。

因此第一版建议的五个核心动作是：

```text
新增
列表与详情
编辑
关键词搜索
开放格式保存或导出
```

“列表”和“详情”可以视为同一阅读能力的两个层次。

### 2. 导出不是一个孤立按钮

真正的需求不是“有导出按钮”，而是：

```text
数据不被系统锁死，可以被其他工具和 AI 继续读取和使用
```

因此需要优先决定底层保存格式，再决定是否额外提供导出操作。

## 十、页面状态

产品不是只有“正常显示”一种状态。至少要考虑：

| 状态 | 含义 | 页面表现 |
|---|---|---|
| 初始状态 | 第一次进入系统 | 显示空状态和新增引导 |
| 正常状态 | 已有记录 | 显示列表和搜索结果 |
| 表单错误 | 必填项为空或格式错误 | 明确提示具体问题 |
| 无匹配结果 | 搜索后没有记录 | 显示没有符合条件的笔记 |
| 保存成功 | 新增或编辑完成 | 列表立即更新并给出反馈 |
| 导出成功 | 文件生成完成 | 明确显示文件位置或下载结果 |
| 附件不可用 | 路径失效或文件缺失 | 显示明确错误，不伪装为正常 |

后续接入 API 时，还需要 loading 和网络 error 状态。

## 十一、验收标准

验收标准必须能被实际操作验证，而不是使用“美观”“好用”“功能完整”等模糊词。

模糊标准：

```text
系统方便 AI 使用，资料管理能力强。
```

可验收标准：

```text
1. 用户填写标题和正文后点击保存，列表中立即出现新笔记。
2. 标题为空时不能保存，并在标题字段附近显示错误提示。
3. 用户输入正文中的任意关键词后，搜索结果能显示对应笔记。
4. 用户打开笔记详情并修改正文，保存后再次打开时显示最新内容。
5. 导出的笔记可以使用普通文本编辑器打开，并保留标题、正文和元数据。
6. 导出的 Markdown 文件可以复制到其他文件夹中继续读取，不依赖原系统数据库。
7. 附件引用失效时，系统明确提示文件不可用。
```

每条验收标准最好包含：

```text
前提 + 操作 + 预期结果
```

## 十二、任务卡和 Prompt 的区别

普通 Prompt 可能只是一次对话指令；工程任务卡应当是可追踪、可审查的工作合同。

一个有效任务卡至少包含：

- 任务目标。
- 授权修改文件。
- 授权新建文件。
- 禁止触碰文件。
- 功能范围。
- 非目标。
- 验收标准。
- 风险和边界情况。
- 在线可检查项。
- 必须本地验证项。

执行端不能根据自己的偏好扩大任务。

## 十三、常见错误

### 错误 1：把三个系统层同时塞进第一版

自动采集、知识库管理和 AI 问答同时开发，会迅速失控。

### 错误 2：功能列表代替需求

“新增、搜索、导出”只是功能，还需要说明格式、流程和验收方式。

### 错误 3：把“导出”理解为临时补救

如果底层数据格式仍然封闭，导出功能很容易丢失附件、元数据和关联关系。

### 错误 4：让 AI 自己决定全部字段

AI 可以建议，但字段最终应由实际研究流程决定。

### 错误 5：只看最终页面，不看 Diff

页面看起来正常，不代表没有超范围改动、硬编码或安全问题。

### 错误 6：把“代码生成成功”当作“项目完成”

未运行、未测试、未验收的代码只能算候选结果。

## 十四、在线审查与本地验证的边界

老师和执行端在线可以检查：

- 文件结构。
- Markdown 和元数据设计。
- 代码逻辑。
- Diff 是否超范围。
- 文档是否完整。
- 静态语法和明显风险。

通常必须本地或部署环境验证：

- 页面是否真正打开。
- 搜索和编辑是否正确。
- 导出的文件是否能在其他工具中打开。
- 附件相对路径是否有效。
- 浏览器控制台是否报错。
- 视觉是否符合预期。

## 十五、本课需要掌握的术语

| 术语 | 本课含义 |
|---|---|
| User | 主要使用者 |
| Problem | 要解决的具体痛点 |
| Scope | 本轮包含的工作 |
| Non-goal | 本轮明确不做的内容 |
| User Flow | 用户完成目标的操作路径 |
| Field | 一条数据中保存的属性 |
| State | 页面或数据当前所处状态 |
| Open Format | 不依赖单一软件即可读取和迁移的数据格式 |
| Portability | 数据迁移到其他系统后仍可继续使用的能力 |
| Acceptance Criteria | 判断任务完成的可执行条件 |
| Task Card | 对 AI 修改范围和交付结果的正式约束 |
