# 第 1 课教学讲义：从业务想法到工程规格

## 一、为什么第一课不是写代码

Vibe Coding 最常见的失败，不是模型不会写代码，而是输入给模型的任务本身不完整：

- 目标不清楚。
- 功能范围不断变化。
- 字段和状态没定义。
- 只描述“长什么样”，没描述“怎么工作”。
- 没有验收标准，最后只能凭感觉判断。

对有明确业务想法、但工程经验不足的人来说，最重要的能力不是背语法，而是把业务语言翻译成工程语言。

```text
模糊想法：做一个研究笔记工具

工程规格：谁使用、解决什么问题、有哪些页面和操作、保存哪些字段、有哪些状态、第一版不做什么、如何验收
```

## 二、业务语言和工程语言的区别

### 业务语言

业务语言通常强调价值和结果：

- 我想集中管理研究笔记。
- 我希望快速找到某个标的的历史观点。
- 我想知道哪些观点已经兑现、失效或仍在跟踪。

这些表达有方向，但 AI 还无法据此稳定实现。

### 工程语言

工程语言必须补充可执行信息：

- 页面有哪些区域。
- 用户可以执行哪些动作。
- 每条数据包含哪些字段。
- 数据有哪些状态。
- 操作成功、失败、为空时如何显示。
- 什么条件下算完成。

业务目标回答“为什么做”，工程规格回答“具体做成什么”。

## 三、一个完整规格的七层结构

### 1. 用户与问题

先明确主要用户，不要一开始试图满足所有人。

示例：

```text
主要用户：投资研究人员
问题：研究观点分散在聊天、文档和表格中，难以按标的、状态和日期回看
```

错误写法：

```text
所有投资人都能使用的智能研究平台
```

它范围过大，也没有指出第一版解决的具体问题。

### 2. 目标

目标应描述第一版带来的可观察变化。

示例：

```text
用户可以在一个页面新增、查看、筛选和编辑研究笔记，并在刷新页面后保留模拟数据。
```

### 3. 范围与非目标

范围决定本次做什么；非目标防止开发过程中不断加需求。

第一版可以做：

- 新增笔记。
- 查看笔记列表。
- 编辑和删除。
- 按标的和状态筛选。
- 本地保存。

第一版不做：

- 登录和多人协作。
- 实时行情。
- AI 自动总结。
- 云数据库。
- 权限管理。
- 真实研究数据导入。

非目标不是永远不做，而是本轮不做。

## 四、用户流程

用户流程描述用户从进入页面到完成目标的步骤。

研究笔记看板的最小主流程可以是：

```text
打开页面
→ 查看现有笔记
→ 点击新增
→ 填写字段
→ 保存
→ 在列表中看到新记录
→ 使用筛选找到记录
→ 编辑状态或内容
```

流程应覆盖正常路径，也要考虑失败和空状态。

## 五、数据字段与数据模型

字段不是越多越专业。第一版只保留能支持核心流程的字段。

建议字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `id` | 唯一标识 | 区分每条记录 |
| `createdAt` | 日期时间 | 记录创建时间 |
| `title` | 文本 | 笔记标题 |
| `asset` | 文本 | 标的或主题 |
| `thesis` | 长文本 | 核心观点 |
| `catalyst` | 长文本 | 预期催化剂 |
| `risk` | 长文本 | 主要风险 |
| `status` | 枚举 | 跟踪中、已验证、已失效 |

字段设计需要回答：

- 是否真的用于页面展示、筛选或业务判断？
- 是否有明确类型？
- 是否允许为空？
- 是否存在固定选项？

## 六、页面状态

产品不是只有“正常显示”一种状态。至少要考虑：

| 状态 | 含义 | 页面表现 |
|---|---|---|
| 初始状态 | 第一次进入页面 | 显示空状态和新增引导 |
| 正常状态 | 已有记录 | 显示列表和筛选结果 |
| 表单错误 | 必填项为空或格式错误 | 明确提示具体问题 |
| 无匹配结果 | 筛选后没有记录 | 显示“没有符合条件的笔记” |
| 保存成功 | 新增或编辑完成 | 列表立即更新并给出反馈 |
| 删除确认 | 防止误删 | 删除前进行确认 |

后续接入 API 时，还需要 loading 和网络 error 状态。

## 七、验收标准

验收标准必须能被实际操作验证，而不是使用“美观”“好用”“功能完整”等模糊词。

模糊标准：

```text
页面简洁，笔记管理方便。
```

可验收标准：

```text
1. 用户填写标题、标的、核心观点和状态后点击保存，列表中立即出现新记录。
2. 标题为空时不能保存，并在标题字段附近显示错误提示。
3. 刷新浏览器后，已保存的模拟记录仍然存在。
4. 选择“已失效”状态后，列表只显示该状态记录。
5. 删除记录前必须二次确认，确认后记录消失。
```

每条验收标准最好包含：

```text
前提 + 操作 + 预期结果
```

## 八、任务卡和 Prompt 的区别

普通 Prompt 可能只是一次对话指令；工程任务卡应当是可追踪、可审查的工作合同。

一个有效任务卡至少包含：

- 任务目标。
- 授权修改文件。
- 授权新建文件。
- 禁止触碰文件。
- 功能范围。
- 非目标。
- 验收标准。
- 风险和边界情况。
- 在线可检查项。
- 必须本地验证项。

执行端不能根据自己的偏好扩大任务。

## 九、本课案例：研究笔记看板 MVP

本课不直接设计完整基金平台，而是选取一个低风险、可独立运行的内部工具模块。

理由：

1. 与真实研究工作接近，学习结果可迁移。
2. 数据结构明确，适合训练字段、状态和流程。
3. 第一版可以完全使用模拟数据和本地存储。
4. 后续可以自然扩展到筛选、统计、数据库、权限和部署。
5. 不需要接触真实账户、客户或交易接口。

## 十、常见错误

### 错误 1：功能列表代替需求

“新增、删除、筛选”只是功能，还需要说明用户、字段、流程和验收方式。

### 错误 2：第一版就设计完整平台

登录、权限、AI、实时数据、移动端和云数据库同时进入，会让学习目标失焦。

### 错误 3：让 AI 自己决定所有字段

AI 可以建议，但字段最终应由业务目标决定。

### 错误 4：只看最终页面，不看 Diff

页面看起来正常，不代表没有超范围改动、硬编码或安全问题。

### 错误 5：把“代码生成成功”当作“项目完成”

未运行、未测试、未验收的代码只能算候选结果。

## 十一、在线审查与本地验证的边界

老师和执行端在线可以检查：

- 文件结构。
- 代码逻辑。
- Diff 是否超范围。
- 文档是否完整。
- 静态语法和明显风险。

通常必须本地或部署环境验证：

- 页面是否真正打开。
- 按钮和键盘交互是否正确。
- 响应式布局是否正常。
- 浏览器控制台是否报错。
- 数据刷新后是否保留。
- 视觉是否符合预期。

## 十二、本课需要掌握的术语

| 术语 | 本课含义 |
|---|---|
| User | 主要使用者 |
| Problem | 要解决的具体痛点 |
| Scope | 本轮包含的工作 |
| Non-goal | 本轮明确不做的内容 |
| User Flow | 用户完成目标的操作路径 |
| Field | 一条数据中保存的属性 |
| State | 页面或数据当前所处状态 |
| Acceptance Criteria | 判断任务完成的可执行条件 |
| Task Card | 对 AI 修改范围和交付结果的正式约束 |
