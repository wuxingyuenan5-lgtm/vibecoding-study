# 第 1 课教学讲义：从大愿景到可执行小任务

## 一、本课真正要解决的问题

学习者已经有三个真实而且规模很大的方向：

1. 将幕布笔记全面迁移到 Obsidian，并逐步建设 AI 投研知识库。
2. 继续开发 `wuxingyuenan5-lgtm/news-stugy`，完成信息自动搜集、整理和研究日报。
3. 持续维护 `wuxingyuenan5-lgtm/Platform_Experiment` 基金与交易研究平台。

这些方向都是真实需求，但都不能直接变成一条给 AI 的执行指令。

例如下面这句话不能作为工程任务：

```text
帮我把投研全面平台化和 AI 化。
```

它缺少：

- 当前只处理哪个产品。
- 只推进哪个阶段。
- 允许修改哪些文件。
- 输入和输出是什么。
- 明确不做什么。
- 如何验证完成。
- 出错时如何回滚。

因此，本课的核心能力不是写代码，而是建立正确的任务层级。

## 二、必须区分的五个层级

### 1. 长期愿景 Vision

长期愿景描述未来希望形成的整体能力。

学习者当前愿景可以表述为：

```text
形成平台化、自动化、AI 驱动的个人投研工作体系。
```

愿景用于决定方向，不用于直接开发。

### 2. 产品项目 Product

产品项目是承载愿景的独立系统。

当前至少有三个不同产品或项目：

```text
A. Obsidian / AI 投研知识库
B. news-stugy 信息自动搜集整理
C. Platform_Experiment 基金与交易研究平台
```

它们可以相互连接，但不能在同一任务中混写。

### 3. 阶段目标 Milestone

阶段目标是一个产品在一段时间内要形成的完整能力。

例如，知识库产品可以拆成：

```text
阶段 1：确定 Markdown、YAML、目录和附件规范
阶段 2：完成少量幕布样例迁移
阶段 3：建立检索和索引
阶段 4：接入 AI 问答并提供引用
阶段 5：扩大迁移规模、备份和长期维护
```

`news-stugy` 可以拆成：

```text
阶段 1：稳定 RSS / JSON 输入和日报生成
阶段 2：扩大官方来源并提高去重和事件融合质量
阶段 3：完善调度、日志和失败恢复
阶段 4：接入 LLM Analyzer
阶段 5：建立质量评估和人工复核闭环
```

阶段目标仍然可能太大，通常不能一次提交完成。

### 4. 课程练习 Exercise

课程练习只服务于一个学习目标。

例如学习 YAML 时，可以只做：

```text
读取三份 Markdown 文件的 YAML 元数据
→ 检查必填字段
→ 输出错误报告
```

这个练习不需要立即迁移全部幕布笔记，也不需要接入 Obsidian 插件或 AI。

### 5. 单次执行任务 Task

单次执行任务是交给 AI 执行端的最小正式工作合同。

例如：

```text
只在指定练习目录中新增：
- 3 份 Markdown 样例
- 1 个 Python 校验脚本
- 1 份测试文件

脚本只校验 title、createdAt 和 tags。
不接入 Obsidian，不读取真实幕布数据，不修改任何产品仓库。
```

这才是可执行、可审查、可回滚的任务。

## 三、五个层级之间的关系

```text
长期愿景
  ↓ 选择一个产品
产品项目
  ↓ 确定当前阶段
阶段目标
  ↓ 提取需要学习的能力
课程练习
  ↓ 切成一次可审查改动
执行任务
```

常见错误是从“长期愿景”直接跳到“执行任务”。

结果通常是：

- AI 自己补全大量未说明的需求。
- 一次创建很多目录和依赖。
- 修改范围失控。
- 功能看起来很多，但无法验证。
- 旧功能被意外破坏。
- 后续对话越来越长，维护成本不断上升。

## 四、能力主线与产品产出线

课程不以“每一节课都必须给真实产品加功能”为目标。

### 能力主线

课程首先训练可迁移能力：

- Git、终端和环境。
- Python、JavaScript 和 Web 基础。
- Markdown、YAML、JSON、SQL 和文件处理。
- API、错误处理和异步任务。
- Debug、测试和 CI。
- 模块、架构、安全和部署。
- LLM、RAG 和 Agent 工作流。

### 产品产出线 A

```text
Obsidian / AI 投研知识库
```

适合承载：

- 开放格式。
- 元数据规范。
- 幕布迁移。
- 附件管理。
- 搜索与索引。
- AI 检索、问答和引用验证。

### 产品产出线 B

```text
news-stugy
```

当前已经存在 RSS / JSON 输入、SQLite 存储、规则分析、去重、事件融合、排序和 Markdown 日报等基础。

适合承载：

- Python 和 CLI。
- RSS、JSON 和 HTML 处理。
- 数据库。
- 幂等、去重和聚类。
- 测试与 GitHub Actions。
- 调度、日志和失败恢复。
- LLM Analyzer 和质量评估。

已有功能不能因为上课而无意义重写。

### 高级案例库

```text
Platform_Experiment
```

该项目已经涉及前端、后端、执行运行时、正式账务、风险、对账、测试和部署等复杂模块。

基础阶段主要用于：

- 阅读目录和权威文档。
- 理解模块边界。
- 看懂复杂 PR 和 Diff。
- 学习幂等、审计、Fail-closed、Kill Switch 等工程思想。
- 分析测试和 Release Gate。

默认不直接修改。真实变更必须使用更严格的授权和验证门槛。

## 五、如何判断一节课是否进入真实产品

可以使用四个问题判断：

### 1. 当前能力是否已经讲清

例如还不理解 SQL、Migration 和事务时，不应该直接修改复杂产品数据库。

### 2. 任务能否独立验收

必须能明确回答：

```text
执行什么操作后，看到什么结果，就算通过？
```

### 3. 失败是否容易回滚

一次任务应使用独立分支和 PR，并尽量避免同时修改多个高风险模块。

### 4. 产品是否真的需要这项改动

不能为了练习某种技术，就把不必要的依赖或框架加入真实产品。

当任一问题回答为“否”时，优先在学习仓库中做独立练习。

## 六、一个合格工程规格的核心部分

### 1. 用户与问题

先说明谁遇到了什么具体问题。

不合格：

```text
做一个更智能的投研系统。
```

较合格：

```text
同时承担研究和交易的个人用户，需要将分散笔记保存为可迁移文件，以便在 Obsidian 和 AI 工具中继续使用。
```

### 2. 目标

目标描述本轮完成后产生的可观察变化。

```text
完成后，三份模拟笔记能够使用统一 Markdown 和 YAML 结构保存，并通过脚本检查必填字段。
```

### 3. Scope

Scope 是本轮明确包含的工作。

```text
- 三份模拟 Markdown
- 一个字段校验脚本
- 对应测试
```

### 4. Non-goal

Non-goal 是本轮明确不做的内容。

```text
- 不迁移真实幕布数据
- 不安装 Obsidian 插件
- 不接入 LLM
- 不处理 PDF 和图片
- 不修改 news-stugy 或 Platform_Experiment
```

Non-goal 不是永远不做，而是当前不做。

### 5. 输入和输出

必须说明系统读取什么、生成什么。

```text
输入：指定目录中的 Markdown 文件
输出：终端中的校验结果和非零错误退出码
```

### 6. 边界与失败状态

例如：

- YAML 缺失。
- title 为空。
- tags 不是列表。
- 文件编码错误。
- 文件名重复。

### 7. 验收标准

验收标准应尽量包含：

```text
前提 + 操作 + 预期结果
```

例如：

```text
当一份 Markdown 缺少 title 时运行校验命令，终端应指出具体文件和缺失字段，并返回失败状态。
```

## 七、如何把任务切小

### 规则 1：一次只训练一个主能力

学习 YAML 时，不同时加入数据库、Web 页面、AI 和部署。

### 规则 2：优先切垂直小闭环

小闭环应包含最少的输入、处理和输出。

```text
读取文件 → 校验字段 → 输出结果
```

比“先完成所有后端，再完成所有前端”更容易验收。

### 规则 3：一次只承担一种主要风险

不要在一个 PR 中同时修改：

- 数据库结构；
- 交易执行；
- 权限；
- 部署；
- 大型 UI 重构。

### 规则 4：限制文件范围

任务卡必须明确：

- 允许新建什么。
- 允许修改什么。
- 禁止触碰什么。

### 规则 5：先定义验证，再写代码

如果无法说明如何验证，说明任务尚未拆清。

## 八、针对三个真实方向的正确拆法

### 例 1：Obsidian / AI 投研知识库

错误任务：

```text
把幕布全部迁移到 Obsidian，并建立 AI 知识库。
```

合理拆分：

```text
任务 A：设计一份模拟笔记的 Markdown / YAML 规范
任务 B：建立三份样例并人工检查 Obsidian 可读性
任务 C：编写小型校验脚本
任务 D：处理附件相对路径
任务 E：只迁移五条脱敏样例
任务 F：建立关键词检索
任务 G：设计 RAG 测试集
```

### 例 2：news-stugy

错误任务：

```text
把所有新闻都自动抓取并用 AI 分析。
```

合理拆分：

```text
任务 A：新增一个官方 RSS 来源并补充解析测试
任务 B：为单个来源失败增加清晰日志
任务 C：为重复 URL 增加幂等测试
任务 D：设计 LLM Analyzer 的结构化输出 Schema
任务 E：使用固定样例评估事实抽取质量
```

### 例 3：Platform_Experiment

错误任务：

```text
顺便优化基金平台的架构和交易功能。
```

合理学习方式：

```text
任务 A：只读绘制前端、后端和 Runtime 的调用关系
任务 B：选择一个 PR，解释其改动范围和风险
任务 C：阅读一项 Release Gate，说明每个测试防什么错误
任务 D：在独立沙盒复现幂等或对账思想
```

只有到高级阶段，才考虑极小的真实仓库修改。

## 九、常见错误

### 错误 1：把真实项目当作课程目录

课程目录用于学习和归档，产品仓库用于真实实现。两者不能混为一体。

### 错误 2：认为没有产品功能产出就是没有学习

理解 Git、终端、Debug、测试和数据模型，往往比多生成一个页面更重要。

### 错误 3：每次都要求 AI 扫描整个大型仓库

应先读取 README、计划、目标模块和相关测试，再查看 Diff。

### 错误 4：使用“大而正确”的需求描述

“平台化”“AI 化”“提高效率”方向正确，但不能直接验收。

### 错误 5：一次改动跨越多个系统层

采集、存储、AI、交易和平台 UI 不应放进同一任务。

### 错误 6：把 AI 的完成报告当作验证

没有真实运行、测试结果或可检查输出时，只能标记为“待验证”。

## 十、在线审查与本地验证

### 在线可以完成

- 阅读文件结构和文档。
- 检查任务卡和 Scope。
- 检查 Diff 是否超范围。
- 阅读代码和测试逻辑。
- 检查明显的安全和架构风险。

### 通常需要本地完成

- 命令是否真正运行。
- 环境和依赖是否正确。
- 页面和交互是否符合预期。
- 文件路径和编码是否正常。
- 数据库和接口是否真实联通。
- 性能、网络和操作系统相关问题。

## 十一、本课术语

| 术语 | 本课含义 |
|---|---|
| Vision | 长期方向，不直接执行 |
| Product | 独立产品或项目 |
| Milestone | 一个阶段形成的完整能力 |
| Exercise | 服务于学习目标的小练习 |
| Task | 一次可执行、可审查、可回滚的改动 |
| Scope | 本次明确包含的工作 |
| Non-goal | 本次明确不做的工作 |
| Acceptance Criteria | 判断完成与否的可执行标准 |
| Vertical Slice | 同时包含最小输入、处理和输出的小闭环 |
| Product Track | 课程中持续产生真实产品增量的路线 |
| Reference Project | 主要用于阅读和审查训练的复杂项目 |

## 十二、本课结论

学习者的长期产品规划不需要缩小，但每次执行任务必须缩小。

```text
愿景可以很大
阶段必须清楚
课程只练少数能力
任务必须足够小
验收必须有证据
```
