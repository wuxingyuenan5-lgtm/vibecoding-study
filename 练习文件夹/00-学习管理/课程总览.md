# Vibe Coding 课程总览

## 一、课程定位

本课程面向代码基础仍在建立，但已经具备金融研究、交易、产品设计和复杂需求表达能力的学习者。

学习目标不是只完成某一个产品，也不是转型为传统全职程序员，而是建立能够迁移到不同项目的工程基本功：

```text
理解问题
→ 拆解范围
→ 形成规格
→ 约束 AI 执行
→ 看懂项目与 Diff
→ 运行和 Debug
→ 测试与验收
→ 架构与风险控制
→ 长期维护
```

长期方向是平台化、AI 投研化和自动化，但课程不会被任何一个超大型产品绑架。

## 二、学习者真实项目

详细信息见：

```text
练习文件夹/00-学习管理/学习者背景与项目地图.md
```

课程采用：

```text
能力主线
+ Obsidian / AI 投研知识库产品线
+ news-stugy 信息自动整理产品线
+ Platform_Experiment 高级案例库
```

### 产品线 A：Obsidian 与 AI 投研知识库

用于逐步产出：开放格式、幕布迁移、附件管理、检索、AI 问答和引用验证。

### 产品线 B：news-stugy

仓库：`wuxingyuenan5-lgtm/news-stugy`

用于逐步产出：信息采集、RSS / JSON 处理、SQLite、去重、事件融合、日报、自动化、LLM Analyzer 和质量评估。

### 高级案例库：Platform_Experiment

仓库：`wuxingyuenan5-lgtm/Platform_Experiment`

主要用于复杂架构阅读、PR 审查、风险控制、测试、API 契约和工程治理训练，不作为基础课程主要开发产出。

## 三、课程设计原则

1. 基本功优先于产品进度。
2. 每节课只训练少数核心能力。
3. 课堂练习不必都写入真实产品。
4. 独立小练习可以与金融无关，只要更适合讲清概念。
5. 产品增量必须小步、可验证、可回滚。
6. 已经完成的功能不因上课而无意义重写。
7. 真实项目默认先阅读、审查和建模，再决定是否修改。
8. 交易执行、账户、权限、密钥、正式数据和生产部署采用更高风险门槛。

## 五、主线课程规划

课程顺序允许根据掌握情况调整，但不跳过关键基础。

### 阶段 0：协作环境

| 课次 | 主题 | 核心能力 | 主要练习或产出 |
|---|---|---|---|
| 第 0 课 | GitHub 与双聊天协作闭环 | 仓库、提交、推送、老师端与执行端分工 | 已完成 |

### 阶段 1：需求与工程基本功

| 课次 | 主题 | 核心能力 | 主要练习或产出 |
|---|---|---|---|
| 第 1 课 | 从大愿景到可执行小任务 | 用户、问题、Scope、Non-goal、验收标准、任务层级 | 三个真实项目的能力地图与一个微型规格 |
| 第 2 课 | Git、Branch、Commit、PR、Diff 与回滚 | 隔离改动、审查变化、恢复稳定版本 | 一次完整 PR 审查闭环 |
| 第 3 课 | 终端、文件系统、进程、环境变量与依赖 | 理解项目如何被机器运行 | 命令行小实验 |
| 第 4 课 | 编程思维与 Python 基础 | 变量、类型、条件、循环、函数、异常、模块 | 小型文本处理程序 |
| 第 5 课 | Web 基础模型 | HTML、CSS、JavaScript、DOM、浏览器与 HTTP | 独立静态交互页面 |
| 第 6 课 | Debug 与开发者工具 | 复现、日志、断点、最小修复、回归检查 | 故障定位实验 |

### 阶段 2：数据与应用开发

| 课次 | 主题 | 核心能力 | 主要练习或产出 |
|---|---|---|---|
| 第 7 课 | Markdown、YAML、JSON、CSV 与文件 I/O | 数据格式、解析、序列化、编码、路径 | 开放格式转换练习 |
| 第 8 课 | SQL、SQLite、表结构与迁移 | 数据模型、查询、约束、索引、Migration | 小型数据库实验 |
| 第 9 课 | HTTP、API、异步与错误处理 | Request / Response、状态码、超时、重试、loading / empty / error | API 客户端小实验 |
| 第 10 课 | CRUD、表单、状态与前后端契约 | 新增、读取、编辑、删除、校验、接口字段 | 小型应用闭环 |
| 第 11 课 | 测试、质量门槛与 CI | 单元测试、集成测试、Lint、类型检查、GitHub Actions | 建立可重复验证证据 |
| 第 12 课 | 模块、配置、接口与安全边界 | 模块化、依赖方向、配置、Secrets、权限边界 | 小型架构拆分练习 |

### 阶段 3：自动化与 AI 投研

| 课次 | 主题 | 核心能力 | 主要练习或产出 |
|---|---|---|---|
| 第 13 课 | 自动采集流水线 | 定时任务、输入适配、去重、幂等、失败恢复 | `news-stugy` 小范围产品增量 |
| 第 14 课 | Obsidian 知识库与幕布迁移基础 | Markdown、YAML、目录、附件、链接、迁移验证 | 知识库样例与小批量迁移工具 |
| 第 15 课 | LLM API 与结构化输出 | Prompt、Schema、解析、重试、成本和隐私 | 小型 Analyzer |

| 第 17 课 | RAG 与 AI 知识库 | 文档切分、检索、上下文、引用、权限 | Obsidian / AI 知识库增量 |
| 第 18 课 | Agent 工作流 | Tool、Context、Memory、计划、人工审批、失败恢复 | 多步骤研究工作流实验 |

### 阶段 4：复杂平台工程能力

| 课次 | 主题 | 核心能力 | 主要练习或产出 |
|---|---|---|---|

## 六、课程与真实产品的连接节奏

### 基础阶段

以学习仓库中的小练习为主，避免在尚未理解概念时直接修改真实产品。

### 中间阶段

每掌握一组能力，选择一个小任务落到：

- Obsidian / AI 投研知识库；或
- `news-stugy`。

### 高级阶段

使用 `Platform_Experiment` 做架构阅读、风险审查和工程治理训练。只有任务足够小、验证充分并且明确授权时才修改代码。

## 七、每节课的固定文件

每节正式课程至少包含：

```text
README.md
01-教学讲义.md
02-课堂作业.md
03-执行任务卡.md
04-验收清单.md
05-学习日志.md
06-老师评语.md
```

## 八、教学与执行分工

- **老师聊天**：完整授课、维护讲义、设计练习、审查答案和 PR、决定是否进入下一步。
- **AI 执行聊天**：只按批准的任务卡修改授权文件、创建分支和 PR。
- **学习者**：完成关键判断，理解运行结果，需要时在本地运行和反馈。

## 九、默认教学原则

1. 聊天框负责完整教学，Markdown 负责归档。
2. 不能把全部思考外包给 AI。
3. 先写范围、非目标和验收标准，再执行。
4. AI 输出是候选解，运行、Diff、测试和验收证据才是裁决依据。
5. 用户补充的新背景、纠错和长期结论必须同步到对应讲义或学习管理文档。
6. 课程以能力增长衡量，不以短期堆积功能衡量。
