# 学习者背景与项目地图

> 本文件记录长期稳定的学习背景、真实项目和课程边界。老师在规划课程、选择案例和审查任务时默认参考本文件。

## 一、学习者定位

- 代码基础仍在建立，但已经具备较强的金融研究、交易、产品设计和复杂需求表达能力。
- 长期方向是全面转向平台化、AI 投研化和自动化工作流。
- 学习 Vibe Coding 的核心目的，不是只完成某一个产品，也不是转型为传统全职程序员，而是建立扎实、可迁移的工程基本功。
- 重点能力包括：需求拆解、项目结构理解、AI 任务约束、代码和 Diff 审查、运行与 Debug、数据与 API、测试、架构、安全和复杂项目管理。

## 二、真实项目地图

### 1. Obsidian 与 AI 投研知识库方向

当前大量投研笔记保存在幕布。长期计划是：

```text
幕布笔记
→ 迁移到 Obsidian 或其他开放知识库
→ 使用 Markdown、稳定元数据和普通附件目录
→ 支持 AI 检索、总结、问答和研究辅助
```

该方向可以作为课程中的正式产品产出线，但必须拆成小阶段，例如：

1. 知识库目录和元数据规范。
2. 单条笔记 Markdown 模板。
3. 附件相对路径和可迁移性。
4. 小批量样例迁移。
5. 检索与索引。
6. AI 问答、引用和结果验证。
7. 大规模迁移、备份和长期维护。

课程不会一开始承诺完成全部幕布迁移和完整 AI 知识库。

### 2. 信息自动搜集整理项目

仓库：

```text
wuxingyuenan5-lgtm/news-stugy
```

这是一个正在开发的跨资产研究信息整理项目。当前主链路已经包括：

```text
RSS / JSON 输入
→ SQLite 存储
→ 规则分析与事实抽取
→ 精确去重
→ 多来源事件融合
→ 重要性排序
→ Markdown 日报
```

该项目可以作为课程中的另一条正式产品产出线。适合承载：

- Python 和命令行。
- JSON、RSS、HTML 和文件处理。
- SQLite 和数据模型。
- 去重、幂等和事件聚类。
- 测试和 GitHub Actions。
- 定时任务、日志和错误恢复。
- LLM Analyzer、结构化输出和质量评估。

已有功能不会因为课程需要而被无意义重写。课程只选择边界清晰、风险可控的小任务进行迭代。

### 3. 基金与交易研究平台

仓库：

```text
wuxingyuenan5-lgtm/Platform_Experiment
```

这是已经积累较多代码、文档和工程约束的复杂平台项目，涉及前端、平台后端、执行运行时、策略、账户、风控、正式账务、对账和部署等模块。

该项目在课程中的主要定位是：

```text
高级案例库
+ 架构阅读材料
+ PR 和 Diff 审查训练场
+ 风险、测试和工程治理案例
```

它不是基础课程的主要开发产出。默认不为了上课而直接修改该仓库；只有在具备相应能力、任务足够小、风险边界明确且学习者明确授权时，才进入实际变更。

## 三、课程与产品的关系

课程采用“能力主线 + 两条产品产出线 + 一个高级案例库”的结构。

### 能力主线

每节课首先服务于可迁移能力，例如：

- Git、终端和环境。
- Python、JavaScript 和 Web 基础。
- 数据格式、数据库和 API。
- Debug、测试和质量门槛。
- 架构、模块边界和安全。
- LLM、RAG 和 Agent 工作流。

课堂练习可以是独立小项目，不必强行写入真实产品。

### 产品产出线 A

```text
Obsidian / AI 投研知识库
```

### 产品产出线 B

```text
news-stugy 信息自动搜集整理
```

只有当某项能力已经讲清、任务范围适合、验收方式明确时，课程才把它落到上述产品中。

### 高级案例库

```text
Platform_Experiment
```

主要用于强化复杂项目所需的 Vibe Coding 能力，而不是重复建设平台本身。

## 四、默认教学边界

1. 基本功优先于短期产品进度。
2. 不要求每节课都产生真实产品功能。
3. 小练习允许与金融无关，只要更适合解释核心概念。
4. 产品任务必须小步、可验证、可回滚。
5. 不把三个真实方向合并成一个超级项目。
6. 不在学习仓库中复制真实产品的大量代码。
7. 真实项目默认先阅读和审查，再决定是否修改。
8. 涉及交易执行、账户、权限、密钥、正式数据和生产部署时采用更高风险门槛。

## 五、课程成功标准

课程成功不以“完成多少页面或功能”衡量，而以学习者是否能逐步独立完成以下工作衡量：

```text
看懂需求
→ 拆成小任务
→ 选择合理技术方案
→ 约束 AI 修改范围
→ 看懂目录、代码和 Diff
→ 运行、定位和修复问题
→ 写测试和验收证据
→ 维护文档和版本
→ 在复杂项目中控制风险
```
